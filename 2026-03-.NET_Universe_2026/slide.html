왜 사람들이 OpenClaw를 좋아하나

💻 진짜 컴퓨터 접근권

기존 AI는 브라우저 탭 안에 갇혀 있었다. OpenClaw는 내 컴퓨터 위에서 코드 작성·파일 수정·셸 명령 실행을 자율 처리.

🧠 지속적인 기억

이름·선호도·루틴을 학습. 대화가 쌓일수록 더 정확해진다. 데이터는 로컬 마크다운 파일에 저장 — 내 PC 밖으로 나가지 않음.

❤️ 심장박동 (Heartbeat)

명령을 기다리지 않고 스스로 깨어나서 메일 요약·일정 알림·이상 감지를 WhatsApp·Telegram 등으로 먼저 연락.

Agent Framework · 핵심 개념

Agent Framework란 무엇인가

자율적으로 목표를 수행하는 AI 에이전트를 설계·구축·배포·관리하기 위한 소프트웨어 툴킷. 에이전트가 스스로 판단하고 행동하는 전 과정을 지원하는 구조적 토대.

에이전트의 기본 수식

Agent = LLM + Memory + Tools + Orchestration

개발자는 에이전트의 역할, 목표, 허용 도구 범위만 정의하면 된다.
나머지 실행 흐름은 프레임워크가 조율한다.

Agent Framework · 핵심 컴포넌트 ①

오케스트레이션 & 메모리 관리

🎼 오케스트레이션 (Orchestration)

어떤 에이전트를 언제 실행할지 총괄 조율하는 프레임워크의 두뇌. 분기·병렬·조건부 흐름을 동적으로 제어한다.

🧠 메모리 관리 (Memory Management)

에이전트가 단기·장기 기억을 유지해 점점 더 정확한 어시스턴트로 성장하게 한다.

효율 = 검색된 관련 컨텍스트 ÷ 전체 컨텍스트 윈도우 사용량

Agent Framework · 핵심 컴포넌트 ②

도구 통합 & 워크플로우 정의

🔧 도구 통합 (Tool Integration)

에이전트는 주어진 목표에 맞춰 어떤 도구를 언제 사용할지 스스로 판단한다.

🗺️ 워크플로우 정의 (Workflow Definition)

복잡한 태스크를 단계별로 구조화하는 설계도 역할을 한다.

Agent Framework · 추론 메커니즘

ReAct 루프 — 에이전트의 추론-행동 사이클

ReAct(Reasoning + Acting)는 현대 Agent Framework가 채택하는 핵심 추론 루프. 중간 결과를 평가하고 전략을 동적으로 수정한다.

💭 Thought (추론)

목표 달성을 위해 어떤 단계가 필요한지 LLM이 내부적으로 추론

⚡ Action (행동)

추론 결과에 따라 실제 도구를 호출하거나 명령 실행

👁 Observation (관찰)

도구 실행 결과를 관찰하고, 목표 달성 여부를 판단해 루프 지속 여부 결정

Thought → Action → Observation → 반복

환경 변수 로드 & 유효성 검사

API 키를 코드에 하드코딩하지 않고 환경 변수에서 읽는다.

var apiKey  = Environment.GetEnvironmentVariable("OPENAI_API_KEY");
var baseUrl = Environment.GetEnvironmentVariable("OPENAI_BASE_URL");

if (string.IsNullOrWhiteSpace(apiKey))
    throw new ArgumentException("환경 변수 'OPENAI_API_KEY'가 설정되지 않았습니다.");

if (string.IsNullOrWhiteSpace(baseUrl))
    throw new ArgumentException("환경 변수 'OPENAI_BASE_URL'가 설정되지 않았습니다.");

포인트

  • IsNullOrWhiteSpace — null 과 공백 문자열을 동시에 방어한다
  • 실행 초반에 명시적으로 예외를 던져 디버깅을 쉽게 만든다

실전 예제 · 예제 1

ChatClient 생성 → AIAgent 변환 → 실행

// ChatClient 생성
var chatClient = new OpenAIClient(
        new ApiKeyCredential(apiKey!),
        new OpenAIClientOptions { Endpoint = new Uri(baseUrl!) })
    .GetChatClient("gpt-4o-mini");

// AIAgent로 변환
AIAgent agent = chatClient.AsAIAgent(
    instructions: "당신은 친절한 고객 지원 담당자입니다."
);

// 에이전트 실행
var response = await agent.RunAsync("안녕하세요!");
Console.WriteLine(response);

각 줄의 역할

코드역할
Endpoint = new Uri(baseUrl!)커스텀 Base URL 지정 (로컬 모델, 사내 프록시 등 지원)
AsAIAgent(instructions:)ChatClient를 AIAgent로 감싸는 확장 메서드
instructions에이전트의 역할을 정의하는 시스템 프롬프트
RunAsync(string)사용자 메시지 → 추론 → 응답까지 자동 처리

예제 2 — AgentSession으로 대화 맥락 유지하기

'그럼', '그 시간대' 처럼 앞 대화를 참조하는 자연스러운 대화를 구현한다.

목적

AgentSession을 사용해 여러 턴에 걸친 멀티턴 대화를 구현한다.

왜 알아야 하나

기존에는 대화 이력을 직접 쌓고 매번 LLM에 넘겨야 했다. AgentSession을 사용하면 이 과정이 프레임워크 내부에서 자동 처리된다.

세션 생성과 멀티턴 실행

// 에이전트 생성 (name 파라미터 추가)
var agent = ...
    .AsAIAgent(
        instructions: "당신은 친절한 교통 안내 담당자입니다.",
        name: "KTXAgent"
    );

// 세션 생성 — 대화 이력을 담는 컨테이너
AgentSession session = await agent.CreateSessionAsync();

var r1 = await agent.RunAsync("서울에서 부산까지 KTX로 가는데 얼마나 걸리나요?", session);

// "그럼"이 앞 대화(서울-부산)를 참조
var r2 = await agent.RunAsync("그럼 내일 오전 9시에 도착하려면 몇 시 기차를 타야 하나요?", session);

// "그 시간대"가 앞 대화의 출발 시간을 참조
var r3 = await agent.RunAsync("그 시간대 기차의 요금도 알려주세요", session);

CreateSessionAsync() — 대화 이력을 담는 독립 세션 객체 생성

RunAsync(message, session) — session을 넘기면 이전 대화 이력을 자동 포함

예제 3 — 멀티 세션으로 여러 사용자 동시 처리

에이전트 인스턴스는 하나, 세션은 사용자마다 따로.

목적: 하나의 AIAgent 인스턴스를 여러 사용자가 공유하되, 대화 이력은 사용자별로 완전히 분리하는 패턴을 익힌다.

왜 알아야 하나: 실제 서비스에서 사용자마다 에이전트 인스턴스를 만들면 메모리 낭비. 인스턴스 공유 + 세션 분리 패턴이 실무의 정석이다.

실전 예제 · 예제 3

사용자별 독립 세션 생성과 교차 실행

// 에이전트는 딱 한 번만 생성
AIAgent ktxAgent = new OpenAIClient(...)
    .GetChatClient("gpt-4o-mini")
    .AsAIAgent(name: "KTX안내봇", instructions: "...");

// 고객 A 전용 세션
AgentSession sessionA = await ktxAgent.CreateSessionAsync();
var responseA1 = await ktxAgent.RunAsync("서울-부산 소요 시간?", sessionA);
var responseA2 = await ktxAgent.RunAsync("오전 9시 도착하려면?", sessionA);

// 고객 B 전용 세션 (sessionA와 완전히 독립)
AgentSession sessionB = await ktxAgent.CreateSessionAsync();
var responseB1 = await ktxAgent.RunAsync("서울-대전 소요 시간?", sessionB);
var responseB2 = await ktxAgent.RunAsync("그 경로의 첫차 시간은?", sessionB);

// 고객 A가 다시 질문 — sessionA는 고객 A의 앞 대화를 그대로 기억
var responseA3 = await ktxAgent.RunAsync(
    "아까 말한 그 시간대 기차 요금은?", sessionA);

CreateSessionAsync() 를 호출할 때마다 새로운 독립 컨텍스트가 생성된다.

sessionA 에서 실행한 RunAsync 는 오직 sessionA 에 쌓인 이력만 참조한다.

예제 4 — Tool 연동 / 싱글턴 대화 루프

LLM은 현실을 모른다. Tool이 현실과 연결해준다.

목적: 외부 함수를 Tool로 등록해 에이전트가 실시간 데이터를 스스로 가져오도록 한다.

왜 알아야 하나: LLM은 현재 날씨, DB 조회, 외부 API 응답처럼 실시간 외부 데이터를 스스로 알 수 없다. Tool을 등록하면 에이전트가 필요하다고 판단할 때 직접 함수를 호출해 데이터를 가져온다.

실전 예제 · 예제 4

[Description] 으로 Tool 메타데이터 정의

[Description("지정된 도시의 현재 날씨를 조회합니다")]
static string GetWeather(
    [Description("날씨를 조회할 도시 이름")] string city)
{
    return city switch
    {
        "서울" => "맑음, 기온 15도",
        "부산" => "흐림, 기온 18도",
        "제주" => "비, 기온 12도",
        "대구" => "맑음, 기온 17도",
        _      => "정보 없음"
    };
}
  • [Description] 은 LLM이 이 함수를 언제, 어떻게 사용해야 하는지 이해하는 데 쓰인다
  • 함수 자체와 각 파라미터 양쪽 모두에 붙이는 것이 중요하다
  • 설명이 구체적일수록 LLM이 Tool을 정확하게 사용한다

실전 예제 · 예제 4

AIFunctionFactory 로 Tool 등록 & 대화 루프

// Tool 포함 에이전트 생성
var agent = ...
    .AsAIAgent(
        instructions: "당신은 날씨 정보를 제공하는 친절한 어시스턴트입니다.",
        tools: [AIFunctionFactory.Create(GetWeather)]
    );

// 대화 루프
while (true)
{
    Console.Write("You: ");
    string userInput = Console.ReadLine() ?? "";

    if (userInput.ToLower() == "exit") break;

    var response = await agent.RunAsync(userInput);
    Console.WriteLine(response);
}
  • AIFunctionFactory.Create(함수) — 일반 C# 함수를 LLM이 호출 가능한 Tool 객체로 변환한다
  • tools: [...] — 배열 형식으로 여러 Tool을 동시에 등록할 수 있다
  • 이 예제는 session 없이 실행하므로 각 질문은 독립적으로 처리된다

실전 예제 · 예제 5

예제 5 — Tool + Session / 멀티턴 Tool 호출

Tool로 데이터를 가져오고, Session으로 그 결과를 기억한다. 예제 4(Tool)와 예제 2(Session)를 결합한다.

"거기 날씨면 외출하기 좋을까?" 처럼 이전 Tool 호출 결과를 참조하는 자연스러운 대화가 가능해진다.

// Tool + Session 동시 설정
var agent = ...
    .AsAIAgent(
        instructions: "당신은 날씨 정보를 제공하는 친절한 도우미입니다.",
        tools: [AIFunctionFactory.Create(GetWeather)]
    );

var session = await agent.CreateSessionAsync();

// Q1: GetWeather("서울") 호출 → "맑음, 15도" 수신
var response1 = await agent.RunAsync("서울 날씨 알려줘", session);

// Q2: "거기" = session이 기억한 서울 / Tool 재호출 없이 맥락 활용
var response2 = await agent.RunAsync(
    "거기 날씨면 오늘 외출하기 좋을까? 옷차림도 추천해줘", session);

// Q3: GetWeather("부산") 호출 + 서울 결과와 비교
var response3 = await agent.RunAsync(
    "부산은 어때? 서울이랑 비교해줘", session);

RunAsync(message, session) 호출 시 프레임워크가 대화 이력 포함과 Tool 자동 실행을 동시에 처리한다.

// OpenAI
static AIAgent CreateOpenAIAgent() =>
    new OpenAIClient(new ApiKeyCredential(apiKey), options)
        .GetChatClient("gpt-4o-mini")
        .AsAIAgent(name: "OpenAIAgent", instructions: "...");

// Azure — DefaultAzureCredential 자동 선택
static AIAgent CreateAzureOpenAIAgent()
{
    AzureOpenAIClient c = string.IsNullOrWhiteSpace(apiKey)
        ? new AzureOpenAIClient(new Uri(endpoint),
              new DefaultAzureCredential())
        : new AzureOpenAIClient(new Uri(endpoint),
              new ApiKeyCredential(apiKey));
    return c.GetChatClient(deployment)
        .AsAIAgent(name: "AzureOpenAIAgent", instructions: "...");
}

// Anthropic — ApiKey (APIKey 아님)
static AIAgent CreateAnthropicAgent()
{
    AnthropicClient client = new() { ApiKey = apiKey };
    return client.AsAIAgent(model: model,
        name: "AnthropicAgent", instructions: "...");
}

✅ 해당 항목을 세어보세요

  • 사용자 입력이 다양하고 예측하기 어렵다
  • 자연어로 사용자와 대화해야 한다
  • 상황에 따라 동적으로 다른 작업을 수행해야 한다
  • 여러 외부 도구/API를 유연하게 조합해야 한다
  • 다단계 의사결정 과정이 필요하다
  • 완벽한 정확성보다 사용 편의성이 중요하다
  • 약간의 응답 지연(1~2초)을 감수할 수 있다
  • LLM API 호출 비용을 감당할 수 있다
🚀 5개 이상: 강력 추천
🤔 3~4개: 도입 고려
⚙️ 2개 이하: 일반 개발 추천

재무 회계, 의료 진단, 고속 거래, 실시간 게임 로직처럼 100% 정확성 또는 실시간성이 필수인 작업에는 부적합합니다.
